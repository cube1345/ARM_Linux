# 应用开发和驱动开发的分界

嵌入式 Linux 应用开发通常运行在用户态，通过 Linux 暴露出来的接口控制设备，例如 `/dev/ttyS0`、`/dev/i2c-1` 或 `libgpiod`。应用程序一般使用 `open/read/write/ioctl/mmap` 等系统调用，不直接访问寄存器。

驱动开发运行在内核态，负责让 Linux 认识设备，并把硬件能力抽象成用户态可访问的接口。真实驱动可能处理寄存器、中断、DMA、时钟、pinctrl、设备树匹配等内容。本次 `hello_char` 没有真实硬件，只是用内核缓冲区模拟设备数据，所以适合作为驱动入门。

# hello_char 的整体结构

`hello_char.c` 的核心工作是注册一个字符设备，并把用户态文件操作映射到驱动函数。

模块加载后创建 `/dev/hello_char`，用户态访问该节点时进入驱动的回调函数。

```text
insmod hello_char.ko
  -> hello_char_init()
  -> alloc_chrdev_region()
  -> cdev_init()
  -> cdev_add()
  -> class_create()
  -> device_create()
  -> /dev/hello_char 出现

cat /dev/hello_char
  -> hello_open()
  -> hello_read()
  -> copy_to_user()
  -> hello_release()

echo "hello from user" > /dev/hello_char
  -> hello_open()
  -> hello_write()
  -> copy_from_user()
  -> hello_release()

rmmod hello_char
  -> hello_char_exit()
  -> device_destroy()
  -> class_destroy()
  -> cdev_del()
  -> unregister_chrdev_region()
```

# 关键对象

 **`dev_t hello_dev`**  **。** 保存设备号，内部包含主设备号 major 和次设备号 minor。驱动通过 `alloc_chrdev_region()` 动态申请设备号，通过 `unregister_chrdev_region()` 释放。

 **`struct cdev hello_cdev`**  **。** 字符设备在内核中的对象。它把设备号和 `file_operations` 绑定起来。没有 `cdev_add()`，用户态即使有设备号，也无法正确进入驱动回调。

 **`struct class *hello_class`**  **。** 用于创建设备类，再配合 `device_create()` 在 `/dev` 下生成设备节点。本次最终得到的是 `/dev/hello_char`。

 **`struct file_operations hello_fops`**  **。** 这是用户态文件操作到驱动函数的映射表。例如 `.read = hello_read`，表示用户态调用 `read(fd, ...)` 时会进入 `hello_read()`。

 **`hello_buffer`**  **。** 驱动内部的模拟设备数据。`read()` 从这里取数据给用户态，`write()` 把用户态传入的数据放到这里。

 **`DEFINE_MUTEX(hello_lock)`**  **。** 保护 `hello_buffer`。多个进程同时读写字符设备时，如果不加锁，可能出现读到半截数据或写入互相覆盖的问题。

# 初始化函数为什么用 goto

`hello_char_init()` 按顺序申请资源：设备号、cdev、class、device node。任何一步失败，都必须释放前面已经成功申请的资源。内核代码常用 `goto` 做错误回滚，因为它能让正常路径保持清晰，也能避免重复写释放代码。

```text
cdev_add 失败：
  已申请设备号
  -> goto unregister_chrdev
  -> unregister_chrdev_region()

class_create 失败：
  已申请设备号 + 已注册 cdev
  -> goto delete_cdev
  -> cdev_del()
  -> unregister_chrdev_region()

device_create 失败：
  已申请设备号 + 已注册 cdev + 已创建 class
  -> goto destroy_class
  -> class_destroy()
  -> cdev_del()
  -> unregister_chrdev_region()
```

这些标签之间没有 `return` 是故意的。代码会从对应标签向下贯穿执行，确保释放顺序与申请顺序严格相反。

# read 和 write

`hello_read()` 是内核把数据给用户态，所以使用 `copy_to_user()`。它会根据 `*ppos` 文件偏移判断是否读到结尾，读完返回 `0`，用户态工具如 `cat` 会把这视为 EOF。

`hello_write()` 是用户态把数据交给内核，所以使用 `copy_from_user()`。驱动不能直接解引用 `char __user *` 指针，因为这个地址属于用户态，内核必须通过专门的 user copy API 安全访问。

# Makefile 怎么写来着？？？

外部内核模块需要让 Kbuild 进入内核构建目录，同时通过 `M=$(CURDIR)` 告诉内核当前外部模块目录在哪里。`ARCH` 指目标架构，`CROSS_COMPILE` 必须是工具链前缀，最后停在短横线，不要写成完整的 `gcc`。

```makefile
KDIR ?= /home/cube/WorkSpace/Linux/ARM_Linux/build/linux-6.1.44
ARCH ?= arm64
CROSS_COMPILE ?= /home/cube/WorkSpace/Linux/ARM_Linux/host/bin/aarch64-buildroot-linux-gnu-

CC := $(CROSS_COMPILE)gcc
CFLAGS := -Wall -Wextra -O2

obj-m += hello_char.o

all:
	$(MAKE) -C $(KDIR) M=$(CURDIR) ARCH=$(ARCH) CROSS_COMPILE=$(CROSS_COMPILE) modules

app: hello_test

hello_test: hello_test.c
	$(CC) $(CFLAGS) $< -o $@

clean:
	$(MAKE) -C $(KDIR) M=$(CURDIR) ARCH=$(ARCH) CROSS_COMPILE=$(CROSS_COMPILE) clean
	rm -f hello_test
```

本次踩过的典型错误包括：Makefile 命令前用了空格而不是 Tab；文件名写成小写 `makefile` 导致 Kbuild 找不到大写 `Makefile`；`ARCH = arm64` 等号两侧有空格导致 make 解析错误；`CROSS_COMPILE` 使用 `~/...` 导致内核脚本调用 `nm` 失败；用户态测试程序没有指定交叉编译器，结果被本机 `cc` 编成 x86-64。

# ioctl

`read/write` 负责数据流，`ioctl` 负责控制命令。

```c
#ifndef HELLO_CHAR_H
#define HELLO_CHAR_H

#include <linux/ioctl.h>

#define HELLO_IOC_MAGIC 'h'
#define HELLO_IOC_CLEAR   _IO(HELLO_IOC_MAGIC, 0)
#define HELLO_IOC_GET_LEN _IOR(HELLO_IOC_MAGIC, 1, int)

#endif
```

这里的魔数 `'h'` 是 ioctl 命令族 ID，用来区分“这是 hello_char 这一组命令”。完整 ioctl 命令号由方向、魔数、编号和数据大小编码而成。真实驱动通常会检查 `_IOC_TYPE(cmd)`，防止应用把别的驱动命令发到当前设备上。

# 阻塞 read 与 wait queue

`read_queue` 是等待队列，负责挂起等待数据的进程；`data_ready` 是状态位，表示当前缓冲区是否有新数据可以读。

```c
static wait_queue_head_t read_queue;
static int data_ready;

static int __init hello_char_init(void)
{
    init_waitqueue_head(&read_queue);
    data_ready = 0;
    ...
}
```

阻塞读的核心逻辑是：如果当前文件偏移为 0 且没有数据，就进入 `wait_event_interruptible()`。这会让当前进程睡眠，直到 `data_ready` 变为真或进程收到信号。

```c
if (!*ppos) {
    if (!data_ready) {
        ret = wait_event_interruptible(read_queue, data_ready);
        if (ret)
            return ret;
    }
}
```

这里有一个关键点：不能在持有 `mutex` 的时候睡眠等待。否则读进程拿着锁睡着，写进程想拿锁写入并唤醒读进程，就会相互卡住。因此等待动作要放在 `mutex_lock()` 之前。

```c
hello_buffer[len] = '\0';
data_ready = 1;
mutex_unlock(&hello_lock);
wake_up_interruptible(&read_queue);
```

# O_NONBLOCK：让 read 不负责等待

阻塞 read 的缺点是，一个线程调用 `read()` 后会卡在这个设备上。如果应用还要同时处理 UART、socket、按键、定时器等多个 fd，就不能把线程堵死在单个 `read()` 里。

`O_NONBLOCK` 的语义是：如果当前没有数据，`read()` 不等待，立刻返回 `-EAGAIN`。等待这件事交给 `poll()`统一处理。

```c
if (!*ppos) {
    if (!data_ready) {
        if (file->f_flags & O_NONBLOCK)
            return -EAGAIN;

        ret = wait_event_interruptible(read_queue, data_ready);
        if (ret)
            return ret;
    }
}
```

# poll：统一等待多个 fd

`poll()` 用来询问设备当前是否可读、可写或有异常。用户态调用 `poll()` 时，内核会通过 `file_operations` 找到驱动注册的 `.poll` 回调，也就是本驱动里的 `hello_poll()`。

```c
static const struct file_operations hello_fops = {
    .owner = THIS_MODULE,
    .open = hello_open,
    .release = hello_release,
    .read = hello_read,
    .write = hello_write,
    .unlocked_ioctl = hello_ioctl,
    .poll = hello_poll,
};
```

`hello_poll()` 做两件事：先用 `poll_wait()` 把当前进程登记到 `read_queue` 上；再检查 `data_ready`，如果已有数据，就返回 `POLLIN | POLLRDNORM`，告诉用户态这个 fd 当前可读。

```c
static __poll_t hello_poll(struct file *file, poll_table *wait)
{
    __poll_t mask = 0;

    poll_wait(file, &read_queue, wait);

    mutex_lock(&hello_lock);
    if (data_ready)
        mask |= POLLIN | POLLRDNORM;
    mutex_unlock(&hello_lock);

    return mask;
}
```

`poll_wait()` 本身不是长时间睡眠，它主要是把“当前进程应该睡在哪个等待队列上”登记给内核的 poll 框架。真正等待由 `poll()` 系统调用框架完成。等写进程调用 `wake_up_interruptible(&read_queue)` 后，poll 框架会再次调用 `hello_poll()` 检查状态。

```text
用户态 poll()
  -> 内核调用 hello_poll()
  -> poll_wait() 登记 read_queue
  -> data_ready == 0，返回 0，poll 继续等待

另一个进程 write()
  -> 写 hello_buffer
  -> data_ready = 1
  -> wake_up_interruptible(&read_queue)

poll 被唤醒
  -> 再次调用 hello_poll()
  -> data_ready == 1
  -> 返回 POLLIN | POLLRDNORM
  -> 用户态 poll() 返回
  -> 用户态再调用 read()
```
